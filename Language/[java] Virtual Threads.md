# Virtual Threads (Java 25 기준)

## 1. 개요 (Overview)
* **Virtual Threads(가상 스레드)**는 Java 19에서 프리뷰로 등장하여 Java 21에서 정식 기능으로 도입된 후, **Java 25**에 이르러 안정화 및 기능 확장이 이루어진 Project Loom의 핵심 경량 스레드 모델입니다.
* JVM이 직접 스케줄링하고 관리하며, 기존의 플랫폼 스레드(OS 스레드)와 달리 매우 적은 메모리 풋프린트를 가져 단일 애플리케이션 내에서 수백만 개의 스레드를 동시에 생성하고 유지할 수 있습니다.

## 2. 기존 OS 스레드 및 스레드 풀 모델과의 차이점

### 전통적인 플랫폼 스레드 (Platform Threads)
* **1:1 매핑**: Java의 기존 `java.lang.Thread`는 OS의 네이티브 스레드와 1:1로 매핑됩니다.
* **무거운 리소스**: OS 스레드는 생성 및 컨텍스트 스위칭 비용이 매우 높으며, 기본적으로 스레드당 약 1MB의 메인 메모리(Stack)를 점유합니다.
* **스레드 풀의 한계**: 무거운 스레드를 매번 생성할 수 없어 **스레드 풀(Thread Pool)**을 사용하지만, I/O 작업(DB 쿼리, 외부 API 호출 등) 발생 시 OS 스레드 자체가 블로킹(Blocking)되므로 동시 처리량(Throughput)이 스레드 풀 사이즈에 제한을 받습니다.

### 가상 스레드 (Virtual Threads)
* **M:N 매핑**: JVM 내부에서 수많은 가상 스레드(M)가 소수의 캐리어 스레드(Carrier Threads = OS 스레드, N) 위에서 실행되는 구조를 가집니다.
* **초경량 저비용**: 메모리 풋프린트가 수십~수백 바이트 수준으로 극히 작으며, 컨텍스트 스위칭이 OS를 거치지 않고 JVM 힙 영역 메모리 이동만으로 이루어져 매우 빠릅니다.
* **스레드-퍼-리퀘스트(Thread-per-Request)의 부활**: 리소스 제약 탓에 강제되었던 복잡한 비동기(Asynchronous)나 리액티브(Reactive) 프로그래밍 없이, 요청 하나당 가상 스레드 하나를 할당하는 직관적인 동기식 코드로도 압도적인 동시성을 달성할 수 있습니다.

## 3. 논블로킹 I/O 처리에서의 이점 (Project Loom)

* **자동 언마운트(Unmount)**: 가상 스레드 내에서 블로킹 작업(`Thread.sleep()`, 네트워크 소켓 I/O, DB 접근 등)이 발생하면, JVM은 해당 가상 스레드를 캐리어 스레드에서 즉시 언마운트하여 힙(Heap) 메모리에 보관합니다.
* **캐리어 스레드 재사용**: 가상 스레드가 언마운트되어 대기하는 동안, 해당 캐리어 스레드(OS 스레드)는 블로킹되지 않고 즉시 다른 가상 스레드의 작업을 할당받아 처리를 이어갑니다. I/O가 완료되면 대기 중이던 가상 스레드는 다시 사용 가능한 캐리어 스레드에 마운트되어 실행을 재개합니다.
* **논블로킹의 투명성**: 개발자는 `CompletableFuture`나 콜백(Callback) 지옥 없이 전통적인 블로킹 코드를 그대로 작성해도, JVM 내부적으로는 완전한 논블로킹 I/O 및 이벤트 루프와 동일한 효율을 자동으로 얻게 됩니다.

## 4. Java 25 기준 최신 동향 및 개선점

* **모니터 피닝(Monitor Pinning) 최적화**: Java 21 등 초기 버전에서는 `synchronized` 블록이나 JNI 호출 내부에서 블로킹이 발생할 경우 가상 스레드가 캐리어 스레드를 물고 놓지 않는 'Pinning' 문제가 있었습니다. Java 25에서는 JVM 락 메커니즘 개선을 통해 이 한계가 대폭 해소되어, 레거시 라이브러리를 사용할 때의 성능 저하 우려가 크게 줄어들었습니다.
* **구조적 동시성 (Structured Concurrency) & 스코프 값 (Scoped Values) 성숙**: 수백만 개의 가상 스레드 생명주기와 에러 전파, 취소(Cancellation)를 안전하고 체계적으로 관리할 수 있는 구조적 동시성 API가 Java 25에서 완전히 자리잡았습니다. 또한 무거운 `ThreadLocal`을 대체하여 가상 스레드 환경에서 메모리 누수를 방지하는 `ScopedValue`의 활용이 표준화되었습니다.
* **관측성(Observability) 및 진단 도구 완성**: 수백만 개의 스레드가 생성되는 환경에 맞게 JFR(Java Flight Recorder) 및 스레드 덤프(jcmd 등) 분석 도구가 고도화되어, 가상 스레드들을 그룹 단위로 필터링하고 데드락이나 병목 구간을 시각화하는 기능이 완벽히 지원됩니다.

## 5. 면접 핵심 요약

1. **개념**: OS 스레드와 분리되어 JVM이 직접 관리하는 초경량 스레드.
2. **해결하는 문제**: 기존 스레드 풀의 컨텍스트 스위칭 오버헤드와 블로킹 I/O 병목을 해결하고, 리액티브 프로그래밍의 복잡성을 낮춤.
3. **동작 원리**: 가상 스레드가 I/O 대기 시 캐리어 스레드에서 분리(Unmount)되고, 작업 완료 시 다시 결합(Mount)되어 OS 스레드 낭비를 최소화함.
4. **Java 25 관점**: 초기 `synchronized` 피닝 문제의 상당한 개선과 구조적 동시성 생태계의 완성으로, 엔터프라이즈 환경에서의 완전한 실무 도입 장벽이 허물어짐.
